# Sentry SDK compatibility matrix

Trackdown implements the Sentry envelope wire protocol so existing Sentry SDKs work without modification. This page tracks what's verified working today — updated honestly as coverage grows, including known gaps. "Verified" means: driven by the real SDK (not a hand-built request) against a running Trackdown instance — both in automated tests (see `internal/ingest/sdk_compat_test.go`, `internal/ingest/sentrygo_e2e_test.go`, `internal/protocol/envelope_test.go`, and the fixture generators under `tools/genfixtures*`) and, for the initial verification of each SDK, against the actual compiled `trackdown.exe` binary (see `tools/e2echeck*`).

## Required client-side compatibility notes (things a self-hoster or SDK user needs to know)

- **JS, Python, .NET, and Java SDKs all require a *numeric* project ID in the DSN.** They validate the DSN client-side and reject a non-numeric project-id path segment with a hard error — confirmed directly against each SDK, not inferred from docs. `sentry-go` has no such restriction and accepts an arbitrary string. **Trackdown itself treats project IDs as opaque strings and has no numeric requirement** — but if you're using any SDK besides Go, choose a numeric project ID (e.g. `12345`) when setting up a project, or the SDK will refuse to initialize at all.
- **`sentry-sdk` (Python), the .NET SDK, and the Java SDK all gzip-compress every envelope by default, with no size threshold.** Confirmed for Python by reading its `transport.py` directly (`compresslevel=9`, unconditional unless disabled); confirmed for .NET and Java by inspecting a real captured fixture's `Content-Encoding` header. Trackdown's ingest endpoint decompresses `Content-Encoding: gzip` bodies (see `internal/ingest/ingest.go`'s `readEnvelopeBody`) — this was a real gap found and fixed during verification, not a hypothetical: **before this fix, every Python SDK event would have silently failed to parse**, and the same would have been true for .NET/Java had they been verified first. `Content-Encoding: br` (Brotli) is explicitly rejected with a `400` (not supported); only `gzip` and no encoding are handled today.
- **The .NET SDK sends `CaptureMessage` text under a top-level `"logentry"` field (`{"message": "..."}`), not the `"message"` field the other four SDKs use.** Both are real, current Sentry protocol field names for the same concept. `internal/protocol.ParseEvent` normalizes `logentry` into the same `Event.Message` field `"message"` populates, so this is transparent to everything downstream — but it's a genuinely SDK-specific quirk, confirmed against a real captured fixture, not something inferable from the other SDKs' behavior.
- **The Java SDK omits the top-level `"level"` field entirely for exception events** (the other four SDKs all set it explicitly to `"error"`). The Sentry protocol spec documents `level` as defaulting to `"error"` when absent — `internal/protocol.ParseEvent` now applies this default itself; before this fix, a Java exception event would have silently stored an empty level.
- **The .NET SDK sends `"threads"` wrapped as `{"values": [...]}`**, matching the documented protocol shape, whereas `sentry-go`/`@sentry/node`/`sentry-sdk` all send a bare array — the same bare-vs-wrapped split `internal/protocol` already handled for `"exception"`, now also handled for `"threads"` (`ThreadValues`).
- **The .NET SDK sends a numeric thread `"id"`** (a real managed thread ID, e.g. `2`) where `sentry-go` sends a string — the Sentry protocol spec allows either. `internal/protocol.Thread.ID` is now a `FlexString` that accepts both forms.
- **Constructing an exception object without throwing it never populates a stack trace**, in both .NET and Java (and this is worth remembering if writing more fixture-capture tooling for either) — `tools/genfixtures-dotnet` and `tools/genfixtures-java` both throw/catch rather than just `new`-ing the exception, to capture real, non-empty stack frames.

## Known gaps (honest, as of this writing)

- Only envelope items of `type: "event"` are persisted. Attachments, transactions, sessions, and client reports are accepted (the envelope isn't rejected) but silently dropped.
- The legacy `/api/{id}/store/` endpoint (used by older SDK versions) is not implemented — only the modern `/envelope/` endpoint.
- **Envelope wire-format variance across SDKs, all confirmed and handled:**
  - `sentry-go` sends `exception`, `threads`, and `breadcrumbs` as **bare JSON arrays**; `@sentry/node`, `sentry-sdk` (Python), and the Java SDK send `exception`/`breadcrumbs` **wrapped as `{"values": [...]}`**; the .NET SDK wraps `threads` the same way too. Trackdown's parser (`internal/protocol/event.go`) accepts both shapes for both fields (`ExceptionValues`, `ThreadValues`).
  - Item headers: `sentry-go`, `sentry-sdk` (Python), the .NET SDK, and the Java SDK all send an explicit `length` on the event item; `@sentry/node` does not (implicit-length item, ending at the next newline). Both forms are handled.
  - Timestamps: `sentry-go` and `@sentry/node` send a numeric Unix timestamp (with fractional seconds); `sentry-sdk` (Python), .NET, and Java send an ISO8601 string. Both forms are handled — and, critically, both are normalized to UTC before storage (see `CLAUDE.md`'s hazard note on the CI-breaking bug this fixed).
  - Envelope header `sdk` field: present for `sentry-go`, `@sentry/node`, .NET, and Java; **absent** for `sentry-sdk` (Python) — Python only puts SDK info on the event payload itself, not the envelope header. Code that wants SDK identity must read it from the event, not assume it's always on the envelope header.
  - Message text: `sentry-go`/`@sentry/node`/`sentry-sdk`/Java all send it under the top-level `"message"` field; .NET sends it under `"logentry"` instead. Both are normalized into `Event.Message`.
  - `"level"`: `sentry-go`/`@sentry/node`/`sentry-sdk`/.NET all set it explicitly; the Java SDK omits it for exception events, relying on the spec's documented `"error"` default, which `ParseEvent` now applies.
  - Thread `"id"`: a string for `sentry-go`, a number for .NET — `Thread.ID` (`FlexString`) accepts either.
- **`@sentry/node` chained-exception (`Error(..., { cause })`) linking has not actually been verified** — the fixture-capture script disabled `defaultIntegrations` (including the `LinkedErrors` integration that walks `.cause`) for a leaner, more deterministic capture, so the captured fixture only has one exception entry despite being constructed with a cause. `sentry-go` and Python's chained-exception behavior (`raise X from Y`) **are** verified with real 2-entry chains. This is a real, narrow gap, not a documentation oversight — treat Node cause-chaining as unverified until a fixture is captured with default integrations enabled.
- Breadcrumbs and tags have parser support (`internal/protocol/event.go`'s `BreadcrumbList`/`Tags` types, both dual-form) but no SDK has been verified actually sending them yet — none of the fixture-capture scripts add breadcrumbs or tags to the events they emit.
- JS sourcemap symbolication is not implemented. `@sentry/node` stack traces render with minified line/column positions rather than original source locations — an honest, shippable state, not a bug: resolving this is explicitly deferred past this v1 .
